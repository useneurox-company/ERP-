## ПРАВИЛО ФАЗОВОГО ТЕСТИРОВАНИЯ (КРИТИЧНО!)

> **НЕ УХОДИТЬ ДАЛЕКО БЕЗ ПРОВЕРКИ!**

### Принцип:
Каждую фазу разработки, где можно сделать фронт/UI — ДЕЛАЙ И СРАЗУ ТЕСТИРУЙ!

### Алгоритм разработки:
```
1. Написал кусок кода (backend/frontend)
2. ЕСЛИ можно показать в UI → СДЕЛАЙ UI
3. СРАЗУ протестируй эту часть
4. Работает? → Следующая фаза
5. Не работает? → Исправь СЕЙЧАС, не иди дальше!
```

### Почему это важно:
- Чем дальше уходишь — тем сложнее искать баг
- Маленькие проверки = быстрые исправления
- Видишь результат сразу = понимаешь правильно ли идёшь
- Накопленные ошибки = лавина проблем

### Примеры:
| Фаза | Что делать |
|------|------------|
| Создал API endpoint | Сразу проверь curl'ом |
| Добавил кнопку | Сразу кликни, проверь action |
| Написал форму | Сразу заполни, отправь |
| Добавил модуль бота | Сразу отправь команду в бот |

**ПРАВИЛО: Не больше 1-2 фаз без визуальной проверки!**

## ПРОВЕРКА КОНСОЛИ БРАУЗЕРА (КРИТИЧНО!)

### Почему это важно:
- Консольные ошибки = баги которые пользователь не видит, но они ломают функционал
- Ошибки сети = ресурсы не загрузились, API не работает
- Warnings = потенциальные проблемы, deprecated код

### Типы ошибок для отслеживания:
| Тип | Критичность | Описание |
|-----|-------------|----------|
| `error` | КРИТИЧНО | JavaScript ошибки (TypeError, ReferenceError и т.д.) |
| `pageerror` | КРИТИЧНО | Необработанные исключения |
| `network-error` | КРИТИЧНО | Не загрузились ресурсы (404, CORS, timeout) |
| `warn` / `warning` | ВАЖНО | Предупреждения, deprecated API |
| `log` / `info` | INFO | Информационные сообщения (обычно ОК) |

### Как проверять консоль через ScreenCreate:

#### Быстрый тест (рекомендуется):
```bash
curl -X POST http://localhost:3500/test/quick -H "Content-Type: application/json" \
  -d '{"url":"http://localhost:7000"}'
```
Ответ содержит:
- `consoleClean: true/false` — есть ли ошибки
- `console.errorCount` — количество ошибок
- `console.errors` — массив ошибок с деталями

#### Получить все логи консоли:
```bash
curl -X POST http://localhost:3500/test/console
```

#### Получить только ошибки:
```bash
curl -X POST http://localhost:3500/test/console-errors
```

#### Очистить логи перед новым тестом:
```bash
curl -X POST http://localhost:3500/test/console-clear
```

### Алгоритм проверки консоли:

```
1. ПЕРЕД изменениями:
   - Запустить /test/quick на страницу
   - Сохранить baseline ошибок (если есть)

2. ПОСЛЕ изменений:
   - Запустить /test/quick снова
   - Проверить consoleClean === true
   - Если появились НОВЫЕ ошибки → ИСПРАВИТЬ!

3. КРИТЕРИЙ УСПЕХА:
   - consoleClean: true (нет ошибок)
   - warningCount === 0 (желательно)
   - Новых ошибок нет
```

### Типичные ошибки и как исправлять:

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `TypeError: Cannot read property 'x' of undefined` | Обращение к null/undefined | Добавить проверку `if (obj && obj.x)` |
| `ReferenceError: x is not defined` | Переменная не объявлена | Объявить переменную или исправить опечатку |
| `Failed to load: 404` | Файл не найден | Проверить путь, создать файл |
| `CORS error` | Запрос на другой домен | Настроить CORS на сервере |
| `SyntaxError` | Ошибка синтаксиса | Проверить JSON, JS код |

## BROWSERTESTKIT (порт 3600) - Расширенное тестирование

BrowserTestKit - REST API для браузерного тестирования. Порт: 3600

### Быстрый старт:
```bash
# Запустить BrowserTestKit

# Скриншот

curl -X POST http://localhost:3600/end
```
